# Trix Sudoku stress-test set.
#
# The puzzles in this file are too heavy for the default catalog
# experience (the original three exceeded a 15-second solve budget
# during catalog preparation; the 2026-06-05 additions are in the
# 13-14 s band; the anti-brute-force grid solves in ~3h20m -- see
# its entry).  They are shipped as a separate file (rather than the
# default catalog) so the default run stays snappy; this file is
# opt-in.
#
# The solver in examples/sudoku.trx is naive backtracking (no
# constraint propagation, no most-constrained-variable ordering).
# A production solver that added arc consistency and naked-pair
# elimination would crack these in milliseconds.  That is the
# point: these are here to demonstrate where a plain backtracker
# meets its ceiling, not to advertise a solver flaw.
#
# To run ANY stress test, copy its nine rows into
# examples/sudoku-input.txt and run sudoku.trx as usual -- the
# file-override mechanism solves just that single puzzle.  Expect
# runs of 30 seconds to HOURS; the live marching-ants display
# (status line every ~2 s, speculative grid every ~10 s) shows the
# search thinking the whole way, and ctrl-C is always safe.
#
# The 17-clue anti-brute-force grid has 64 empty cells, which
# exceeds the default 64 save-levels (one nested choice level per
# empty cell + the per-puzzle wrapper): run it with
# --save-depth=200.  The other entries fit the default.
#
# Sources: Project Euler problem 96 grids that exceeded the budget
# (PE96 has an exact duplicate: Grid 41 == Grid 09.  Only Grid 09
# is kept below.), plus the canonical anti-brute-force grid from
# Wikipedia's "Sudoku solving algorithms" article -- a 17-clue
# puzzle constructed specifically so ascending-value left-to-right
# backtrackers (exactly this solver) explore an enormous prefix
# space before the constraints bite.
